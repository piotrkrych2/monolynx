## Critical Project Rules

- **Never run Python commands locally** — always use `docker compose exec app <command>`
- Generate migrations with: `make migration msg="description"`
- Run migrations with: `make migrate`
- Run tests with: `make test` or `docker compose exec app python -m pytest tests/ -v`
- Database name is `open_sentry` (historical, kept for backwards compatibility)
- Test database is `open_sentry_test`
- All models use UUID primary keys (no auto-increment)
- PostgreSQL image is `pgvector/pgvector:pg16`

## Alembic Migration Rules

- **Always review autogenerated migrations** — they may include tables from previous migrations if the DB was empty at generation time
- Each migration must only contain its own new/changed tables
- Alembic is configured for async via `asyncio.run()` in `env.py`
- After generating a migration, read the file and verify it contains only the intended changes
- Remove any extraneous table creates/drops that don't belong
- Test migrations by running them: `make migrate`
- When adding columns, consider if they need defaults for existing rows
- For JSONB columns, default to `server_default=text("'{}'::jsonb")`

## SQLAlchemy & Query Optimization

- All database access is async (AsyncSession)
- Use `select()` style queries (SQLAlchemy 2.0 syntax)
- Pagination pattern: count with `func.count()`, then LIMIT/OFFSET
- `event_count` on Issue is denormalized — avoid COUNT(*) queries
- After `db.rollback()`, always re-query objects before passing to Jinja2 templates
- Look for N+1 query patterns and suggest `selectinload()` or `joinedload()`
- When analyzing slow queries, check for missing indexes on filter/join columns
- Soft delete pattern: `is_active = False`, must be filtered in all queries

## pgvector / RAG Search

- Extension: pgvector with HNSW index, cosine similarity
- Model: OpenAI `text-embedding-3-small`
- Chunking: ~500 tokens per chunk with overlap, via tiktoken
- WikiEmbedding model stores chunk embeddings
- Graceful degradation when `OPENAI_API_KEY` not set
- Services: `services/embeddings.py` for embedding generation and search
- Backfill command: `make backfill-embeddings`
- When modifying embedding dimensions or models, consider migration of existing data

## Neo4j Graph Database

- Used by Connections module (`services/graph.py`)
- Node types: File, Class, Method, Function, Const, Module
- Edge types: CONTAINS, CALLS, IMPORTS, INHERITS, USES, IMPLEMENTS
- Data isolated per project via `project_id` property
- Graceful degradation when `ENABLE_GRAPH_DB=false` — `is_enabled()` check, try/except, None fallback
- Async driver singleton pattern
- Always include `project_id` in Cypher queries for data isolation

## Index Strategy

- Check existing indexes before suggesting new ones
- Consider composite indexes for common query patterns
- For JSONB columns, consider GIN indexes if queried frequently
- pgvector HNSW index parameters affect recall vs speed tradeoff
- UniqueConstraint on (project_id, fingerprint) for Issue model

## Quality Assurance

1. After creating migrations, read the generated file to verify correctness
2. Run `make migrate` to test migration applies cleanly
3. Run relevant tests: `docker compose exec app python -m pytest tests/ -v`
4. Run `make lint` to check for type errors in model changes
5. For query changes, consider impact on existing pagination and filtering

## Output Standards

- When suggesting indexes, estimate the impact on read vs write performance
- For migrations, include both upgrade and downgrade paths
- When optimizing queries, show before/after with explanation
- All UI text and comments should be in Polish (project convention)
